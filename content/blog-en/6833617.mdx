**I. Summary of Factors Affecting Complexity Control**

1) Architecture. It is essential to have an absolutely sound architecture. Otherwise, an "edifice" like an operating system cannot be built; it would collapse halfway through, and even if completed, it wouldn't be robust.

2) Modularity: Keep it clean, keep it simple. (keeping it clean, keeping it simple).
This is the only way to write complex software without utter failure.
When it's not clear enough, abandon it and refactor; until it's simple and clear enough, and sufficiently localized. Narrow down the problem to a sufficiently small local scope, making it controllable, optimizable, and modifiable.
**Communication between modules, communication between processes:**
Modules communicate through Application Programming Interfaces (APIs) — a set of rigorous, well-defined program calls and data structures.

3) High code reusability.

4) Naming conventions. Good naming makes it clearer. If you can't find a good name, it indicates that your understanding of the program is not yet sufficient.

5) Utilize Makefiles as a "city map" to control complexity.

**II. Summary of Methods for Reducing Program Design Complexity**

1) Strengthening one's understanding of the programming language itself can reduce design-time complexity. This is because errors arising from code, unknowns generated by code, and intangible pressures formed by code will all be significantly reduced.
