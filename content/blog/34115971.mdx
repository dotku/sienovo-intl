## 

段错误

所谓的段 [错误](http://baike.baidu.com/view/120927.htm)就是指访问的 [内存](http://baike.baidu.com/view/1082.htm)超出了系统所给这个程序的内存空间，通常这个值是由gd tr来保存的，他是一个48位的寄存器，其中的32位是保存由它指向的 gdt表，后13位保存 相应于gdt的下标，最后3位包括了程序是否在内存中以及程序的在 [cpu](http://baike.baidu.com/view/2089.htm)中的运行级别，指向 的gdt是由以64位为一个单位的表，在这张表中就保存着程序运行的 [代码段](http://baike.baidu.com/view/1315853.htm)以及 [数据段](http://baike.baidu.com/view/1005328.htm)的起 始地址以及与此相应的段限和页面交换还有程序运行级别还有内存粒度等等的信息。

#### 英文简介

A segmentation fault (often shortened to segfault) is a particular error condition that can occur during the operation of computer software. In short, a segmentation fault occurs when a program attempts to access a memory location that it is not allowed to access, or attempts to access a memory location in a way that is not allowed (e.g., attempts to write to a read-only location, or to overwrite part of the operating system). Systems based on processors like the Motorola 68000 tend to refer to these events as Address or Bus errors.

Segmentation is one approach to memory management and protection in the operating system. It has been superseded by paging for most purposes, but much of the terminology of segmentation is still used, "segmentation fault" being an example. Some operating systems still have segmentation at some logical level although paging is used as the main memory management policy.

On Unix-like operating systems, a process that accesses invalid memory receives the SIGSEGV signal. On Microsoft Windows, a process that accesses invalid memory receives the STATUS\_ACCESS\_VIOLATION exception.

### 2段错误的常见形式[编辑](http://baike.baidu.com/link?url=q5BY4s6WdLTvMnqZA8yCK-Nel1D8sQhpmfveGuS2ILvgP3cw739Rk5ylvhHDEHkqeGxpQM1kQqwyIAT3Vs8ZjK# "编辑本段")

在 [编程](http://baike.baidu.com/view/3281.htm)中以下几类做法容易导致段 [错误](http://baike.baidu.com/view/120927.htm),基本上是错误地使用 [指针](http://baike.baidu.com/view/159417.htm)引起的。

1) **访问系统数据区**，尤其是往系统保护的 [内存地址](http://baike.baidu.com/view/404417.htm)写 [数据](http://baike.baidu.com/view/38752.htm)最常见就是给一个 [指针](http://baike.baidu.com/view/159417.htm)以0地址。

2) **内存越界**( [数组](http://baike.baidu.com/view/209670.htm)越界， [变量](http://baike.baidu.com/view/296689.htm)类型不一致等)： 访问到不属于你的内存区域。

解决方法：我们在用C/ [C++](http://baike.baidu.com/view/824.htm)语言写程序的时候， [内存管理](http://baike.baidu.com/view/4541016.htm)的绝大部分工作都是需要我们来做的。实际上， [内存管理](http://baike.baidu.com/view/4541016.htm)是一个比较繁琐的工作，无论你多高明，经验多丰富，难免会在此处犯些小 [错误](http://baike.baidu.com/view/120927.htm)，而通常这些错误又是那么的浅显而易于消除。但是手工“除虫”（debug），往往是效率低下且让人厌烦的，本文将就"段错误"这个内存访问 [越界](http://baike.baidu.com/view/392942.htm)的错误谈谈如何快速定位这些"段错误"的语句。

作为一个熟练的 [C/C++](http://baike.baidu.com/view/1107274.htm)程序员，以上代码的bug应该是很清楚的，因为它尝试操作地址为0的 [内存](http://baike.baidu.com/view/1082.htm)区域，而这个内存区域通常是不可访问的禁区，当然就会出错了。我们尝试编译运行它:

#### 利用backtrace和objdump进行分析

#include <execinfo.h>

#include <stdio.h>

#include <stdlib.h>

#include <signal.h>

/\* A dummy function to make the backtrace more interesting. \*/

void

dummy\_function (void)

{

unsigned char \*ptr = 0x00;

\*ptr = 0x00;

}

void dump(int signo)

{

void \*array\[10\];

size\_t size;

char \*\*strings;

size\_t i;

size = backtrace (array, 10);

strings = backtrace\_symbols (array, size);

printf ("Obtained %zd stack frames.\\n", size);

for (i = 0; i < size; i++)

printf ("%s\\n", strings\[i\]);

free (strings);

exit(0);

}

int

main (void)

{

signal(SIGSEGV, &dump);

dummy\_function ();

return 0;

}

### 3几种典型的段错误[编辑](http://baike.baidu.com/link?url=q5BY4s6WdLTvMnqZA8yCK-Nel1D8sQhpmfveGuS2ILvgP3cw739Rk5ylvhHDEHkqeGxpQM1kQqwyIAT3Vs8ZjK# "编辑本段")

1，int main(void){ \[1\]  

char\*s ="hello world";

\*s ='H';

}

被装载时，系统把“hello world” 连同其它的字符串和const型数据放入到内存的只读区。执行时，一个变量s被设为指向该字符串的位置，当再试图向该位置写时，就会产生段错误。

2，

int\*ptr = NULL;

\*ptr =1;

因为该代码只创建了一个空指针，并没有指向一个具体空间，当赋值时，产生段错误。

3，

int main(void){

main();

return0;

}

无限递归，这会导致栈溢出，也会产生段错误。
