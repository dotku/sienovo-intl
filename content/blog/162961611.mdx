## 一、引言

本文基于 ZYNQ UltraScale+ MPSoC 硬件平台开展红外图像处理算法研究与实现。通过为 ZYNQ UltraScale+ MPSoC 移植 Linux 系统、交叉编译 QT 和 OpenCV 应用程序、外接红外摄像头、显示器、网线搭建了适用于红外图像采集及图像处理算法实现的工作环境，首先使用 Linux 内核提供给应用程序的 V4L2 接口实时捕获红外摄像头拍摄的位深为 8bit 的红外图像；调用 imread()方法读取保存在 SD 卡中的位深为 16bit 的高动态原始红外图像。然后在 QT 应用程序中调用 OpenGLES 指令通过内核驱动 GPU 对提出的红外图像增强算法进行加速实现；在ARM 端使用 OpenCV 搭配 C++编程语言将同样的算法也进行实现，最终在显示器上实时显示处理后的结果。

### 2.1 ZYNQ UltraScale+ MPSoC 平台简介

ZYNQ UltraScale+ MPSoC 硬件平台采用核心板搭配底板的模式，核心板主要由ZU3EG+5 个 DDR4+eMMC+1 个 QSPI FLASH 的最小系统组成。ZU3EG 芯片采用处理器系统 Processor System(PS)+可编程逻辑 Programmable Logic(PL)技术将速度高达 **1.2Ghz** 的四核 ARMCortex™-A53、FPGA、GPU 集成在单颗芯片上，同时具备处理器的灵活性和可编程逻辑以及 GPU 的并行计算能力。此外，PS 端挂载了总容量为 **4G** 字节的 4 片DDR4，PL 端挂载了总容量为 **1G** 字节的 1 片 DDR4，使得 FPGA 和 ARM 系统具备独立存储和处理数据的功能，另外 PS 与 PL 之间通过高速接口连接，实现处理器系统和可编程逻辑之间的高速数据交换。

底板为核心板扩展了丰富的外围接口，为交互和采集高速数据、复杂功能的实现提供了可能，同时简化了设计和开发流程。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/03_c64bb7ab8faf4005b39907eaf8d75440.png)

### 2.2 图像处理算法实现的环境搭建

算法实现的环境搭建主要包括 **Linux** **系统**、 **QT** **应用程序**、 **OpenCV** **的移植**以及 **nfs服务和 gdb server** **的搭建**。

#### 2.2.1 Linux 系统移植

由于本研究需要在 QT 和 OpenCV 应用程序中调用 V4L2 接口和 imread() 方法采集红外图像且 OpenGL ES 需要通过内核才能驱动底层 GPU 对红外图像增强算法进行加速实现，因此为 ZYNQ UltraScale+ MPSoC 移植 Linux 系统成为必然。

第一步：配置并导出底层硬件信息。由于本研究仅用到 PS 端的硬件资源，因此只需要配置如图 2.4 所示的 PS 端的 ZYNQ 核即可，包括 DDR4、DP 接口、USB3.0 接口、以 太网接口、SD 卡接口等硬件信息的配置。

第二步：借助 petalinux 工具使用第一步导出的硬件信息完成 BOOT.BIN 文件的制作。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/04_b0e10404b3b94433ae0e8f166c193058.png)

#### 2.2.2 交叉编译 QT、OpenCV 应用程序

在本文中图像处理算法的实现以及 GPU 的调用必须依赖于 QT 和 OpenCV 库函数的支持，为了让 PC 端的软件工程交叉编译的可执行文件能够在 ZYNQ 的嵌入式 Linux 系统中运行，向嵌入式系统移植 QT 和 OpenCV 库函数是必须完成的工作 。

第一步：在 Linux 的文件系统中添加 QT 和 OpenCV 库。

第二步：构建 sdk 。构建好的 sdk.sh 脚本文件中就包含了交叉编译器及各种库函数。

第三步：在安装有 Linux 系统的 PC 机的 QT Creator 中配置 ZYNQ UltraScale+MPSoC 平台的交叉编译环境。

#### 2.2.3 nfs 服务和 gdb server 的搭建

当需要在 ZYNQ 的嵌入式 Linux 系统中运行 PC 端交叉编译生成的可执行文件时，必须将可执行文件拷贝到 SD 卡中，然后再重新启动开发板才能运行相应的程序，非常不方便程序的调试，而 nfs 服务可以在开发板中挂载 PC 机上的工作目录，从而在不使用 SD卡的情况下方便快速地把 PC 端的可执行文件拷贝到开发板中，同时在后续保存处理后的红外图像时，系统默认将其保存在该目录下。本文在 PC 端设置的共享路径为/home/nfs\_work，在确定开发板系统连接网络的情况下使用 mount 命令进行挂载。gdb server 可以通过网络来远程调试应用程序，当在 PC 机的 QT 应用程序中搭建好可远程调试的应用程序后，就可以直接编译和运行红外图像增强算法的程序，不再需要将可执行文件拷贝到 nfs 挂载的共享目录中，更加方便程序的调试。

## 三、高动态原始红外图像的分层增强算法实现及分析

本章将分别在 ZYNQ UltraScale+ MPSoC 硬件平台的 GPU 端和 ARM 端实现上一章节提出的高动态原始红外图像增强算法，在处理之前先将红外图像做归一化处理。算法在 ARM 端使用 OpenCV 搭配 C++ 编程语言实现。

### 3.1 ZYNQ UltraScale+ MPSoC 的 GPU 概述

ARM CPU 占据了移动端 90% 的市场，但它的 Mali GPU 在移动端占有的市场只能算是小众的存在，Mali GPU 在 2006 年被 ARM 收购，成为 ARM 的一个 GPU 事业部

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/08_8530ec8c5cc14bf59d9ffad95a730ff8.png)

ZYNQ UltraScale+ MPSoC 的 GPU 是一款基于 Mali-400 MP2 硬件的 2D 和 3D 加速子系统，一方面拥有着两个独立的 GPU 核心，可以实现更高性能的图形处理，另一方面其功耗相对较低，通常在 200mW~500mW 之间，可以很好地满足大多数嵌入式系统的要求。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/09_d502cd1c09fc4450912cafcc708563c5.png)

### 3.2 QT 框架下 OpenGL ES 2.0 调用 GPU 的相关技术实现

OpenGL ES 2.0 是 OpenGL ES   的一个版本，专门设计用于移动设备和嵌入式系统，一方面引入了基于着色器的可编程渲染管线，使得所有的渲染 工作都需要通过自定义的着色器程序来实现，另一方面引入了纹理和缓冲对象以方便存储和处理图像数据。OpenGL ES 是一组调用嵌入式 GPU 功能的 API 规范，嵌入式 GPU 的硬件开发商需要提供满足 OpenGL ES 规范的实现，这些实现被称为驱动，主要负责将 OpenGL ES 定义的 API 命令翻译为 GPU 指令 。OpenGL ES 调用 GPU 的流程如图 4.1 所示：在 QT 应用程序中调用 OpenGL ES 的接口，继而 OpenGL ES 到硬件驱动中找到接口的实现即 GPU 指令， GPU 根据指令对图像进行处理和显示。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/10_dad43fc9f7fb4c6f9e31a8eecceceadd.png)

#### 3.2.1 OpenGL ES 2.0 的可编程渲染管线

OpenGL ES 渲染管线的处理流程 具体可描述为：

（ 1 ）顶点着色器接收来自于应用程序端的顶点数据并进行运算。

（ 2 ）顶点坐标经图元装配被转换为图元。

（ 3 ）图元通过光栅化被转换成屏幕上的像素并将其传入片段着色器中。

（ 4 ）片段着色器对各个像素点进行颜色填充。可以对图形填充纯色，也可以从纹理图像中采样对应位置的颜色进行填充。

（ 5 ）进行测试与混合。通过测试和混合后，在显示器中显示渲染的结果。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/11_f8ae01493e3d463587ed56f90df7ed36.png)

#### 3.2.2 创建着色器 shader 的技术实现

本文在 QT 中创建 shader 的过程如下：

（ 1 ）在 QT 工程中添加用于编写着色器源码的源文件并编写着色器源码框架，包括定义数据的输入和输出，具体的处理逻辑部分在实现算法时编写。

（ 2 ）使用 QOpenGLShaderProgram 类的 addShaderFromSourceFile() 方法将着色器源码添加到着色器程序中。

（ 3 ）使用 QOpenGLShaderProgram 类的 link() 方法编译、链接着色器源码。

（ 4 ）使用 QOpenGLShaderProgram 类的 bind() 方法绑定着色器程序。

#### 3.2.3 纹理映射的技术实现

#### 3.2.4 “渲染到纹理”的技术实现

在 OpenGL ES 的可编程渲染管线中，默认把帧缓冲区当作渲染目的地，将渲染的结果直接输出到屏幕上，然而大部分的图像处理算法在 GPU 端需要多次渲染才能得以实现，即需要将上一次渲染的结果作为本次渲染的输入。OpenGL ES 允许应用程序创建额外的帧缓冲对象(FBO)以重定向渲染目的地。

FBO 只提供颜色附着、深度附着、模板附着 3 种附着，本身不能用于渲染，只有为其绑定纹理或者渲染缓冲区之后才能作为渲染目的地，绑定为纹理时，OpenGL ES 将执行“渲染到纹理”操作，绑定为渲染缓冲区时，OpenGL ES 将执行“离屏渲染”操作。

![](https://pub-048dcb96257f476697b113fcb5939cb9.r2.dev/blog/162961611/14_6f8dc041110c46db9dd42fea63282711.png)

本文在 QT 应用程序中实现“渲染到纹理”的方法如下：

（ 1 ）调用 QOpenGLFramebufferObject 类的 QOpenGLFramebufferObject() 方法创建一

个帧缓冲对象 (FBO) ，默认会绑定一张纹理。

（ 2 ）调用 QOpenGLFramebufferObject 类的 bind() 方法绑定 FBO ，在绑定期间，所有

的渲染结果都会保存在 FBO 绑定的纹理中。

（ 3 ）当完成一次渲染后，调用 QOpenGLFramebufferObject 类的 release() 方法解绑，

解绑之后，下次渲染结果直接显示在显示器上。

使用“渲染到纹理”技术可以实现多次渲染。在本研究中渲染 N 次，需要创建 N 个顶点着色器、N 个片段着色器和 N-1 个自定义的帧缓冲区。下图为渲染两次的示意图。
